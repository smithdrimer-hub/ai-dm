## 目前功能实现具体程度报告（2026-04-28）

### 总体结论

当前 AI-DM 已经具备“可跑完整局”的基础主持能力：能开场、说明规则、进入讨论、定时公开线索、开放搜证、手动进入投票、进入业主对峙、触发结局并进入复盘阶段。它已经不像纯问答机器人，而是有明确状态机和阶段文本的主持系统。

但如果以“真人 DM 在玩家顺利时少说、玩家卡顿时精准而不剧透地引导”为目标，目前只能算第一版可用，尚未完善。现有引导更多是按阶段和已公开线索数量给固定提示，缺少对玩家真实推理路径、忽略线索、曾经提到但没有追下去的信息进行结构化判断。下一步更推荐先完善已有主持与引导系统，而不是马上做“允许读取陌生脚本”的大改动。

建议评级：

- 开场接待与规则说明：已基本可演示，完成度约 75%。
- 流程主持：已能稳定带完整局，完成度约 70%。
- 智能引导与节奏控制：有雏形，完成度约 45%。
- 游戏时间进度感知：可获取讨论用时，完成度约 55%。
- 真人 DM 式低干预体验：部分满足，完成度约 45%。

### AI-DM 引导功能是否完善

结论：不算完善，但已经有可继续优化的骨架。

现有实现：

- `dm_engine.py` 中有 `stalled_turn_count`、`last_progress_signal`、`consecutive_offtopic_count`，模型输出控制标记里包含 `PROGRESS_SIGNAL`，可识别 `PROGRESS / STALLED / BREAKTHROUGH / UNCHANGED`。
- `STALLED_THRESHOLD = 3`，当连续卡住达到阈值时会调用 `_generate_stall_guidance()`。
- `SILENCE_THRESHOLD = 5`，可根据轮次检查某些角色长时间未发言，并通过 `_check_participation()` 点名。
- `DOMINANCE_THRESHOLD = 4`，可在某个玩家发言明显过多时提醒让其他角色发言。
- 有 `player_claims_history`、`case_memory`、`rolling_summary`，理论上能记录玩家关键陈述、矛盾、未解决问题。
- 有 `_check_contradictions()`、`_detect_mechanical_discussion()`、`_generate_immersion_reminder()`，可做基础矛盾提醒和沉浸感提醒。

主要不足：

- 进度判断主要依赖模型返回的 `PROGRESS_SIGNAL`，没有独立的“推理进度评分器”。如果模型没有稳定输出 STALLED，系统就不一定知道玩家卡住了。
- 冷场引导目前是固定模板：无线索、线索 1、线索 2 三种分支。它没有真正判断“玩家提到过什么但没追问”“玩家完全没注意哪条线索”。
- 提示语有时偏直白，例如“谁能不用钥匙打开橱柜？谁有动机这么做？”已经接近解题方向，不够像真人 DM 的轻推。
- 沉默检测是按回合数，不是按真实静默时间。命令行 `input()` 阻塞期间无法自动感知桌面沉默，只能在下一次输入后判断。
- 参与度检测缺少冷却与去重，后续可能反复提醒同一个沉默角色。
- 对“玩家进度可观时不引导”的约束不够硬。现在只要模型标记 STALLED 或参与度检测触发，就可能插话；缺少基于最近讨论质量的抑制条件。

理想目标差距：

- 理想：玩家正常推进时，DM 只维持秩序和阶段节奏。
- 现状：能沉默观察，但是否沉默强依赖模型控制标记，稳定性还不够。
- 理想：玩家卡顿时，DM 从玩家已说内容、未提线索、错误焦点中挑一个方向轻推。
- 现状：能根据已公开线索数量提示，但没有“线索注意力模型”。
- 理想：提示不明显、不剧透、逐级加压。
- 现状：提示层级不够细，部分提示偏直给。

### 现在能否获取游戏时间进度

结论：能获取“讨论阶段已进行分钟数”，但还不是完整的游戏进度系统。

现有实现：

- `discussion_started_at` 在进入 discussion 时记录 `time.time()`。
- `get_elapsed_discussion_minutes()` 会返回讨论已进行整分钟数。
- `get_status_text()` 会展示“讨论已进行分钟”。
- `poll_timed_events()` 使用该时间：
  - 讨论 10 分钟后开放搜证系统。
  - 10 分钟公开 clue_1。
  - 20 分钟公开 clue_2。
- `/search` 在搜证未开放时会提示距离开放还剩几分钟。

主要不足：

- 时间只覆盖 discussion 阶段，不覆盖开场、投票、对峙、复盘等阶段的分段用时。
- 没有总时长目标，也没有“预计是否能在正常时间内完成”的判断。
- 读档后 `discussion_started_at` 使用保存时的绝对时间戳，理论上会把离线时间也算进讨论时长，需要改为“累计讨论秒数 + 本次载入后开始时间”的方式。
- 没有阶段预算，例如 0-10 分钟自由讨论、10-20 分钟线索推进、20-30 分钟收束投票。
- 没有“进度健康度”指标，例如：已公开线索数量、已发现证据数量、关键问题覆盖率、角色发言覆盖率、距离建议投票时间等。

建议下一步补齐：

- 新增 `phase_started_at`、`discussion_elapsed_seconds_accumulated`、`last_player_input_at`。
- 新增 `get_game_progress_report()`，输出阶段用时、讨论用时、线索进度、搜证进度、投票进度、复盘状态。
- 新增时间预算配置，例如 `target_discussion_minutes=30`、`suggest_vote_after_minutes=25`。
- 在 `/status` 中显示“进度健康度”：正常 / 偏慢 / 卡住 / 建议收束。

### 重要功能一：开场接待与规则说明

目标：系统应完成真人 DM 在游戏开始前的基础主持工作，包括欢迎玩家、说明规则、介绍背景、帮助进入情境。目标不是单纯念说明书，而是让玩家从“准备开始”自然进入“已经在局里”的状态。

当前完成度：约 75%，可演示但还可以更像真人 DM。

已经做好的部分：

- `start_game()` 有欢迎语、剧本名、场景引入、角色代入提醒。
- 明确说明核心规则：非犯人不能说谎、可以使用 NO COMMENT、线索由 DM 公开、最终答案同时公开后录入。
- `_handle_opening_rules_confirmation()` 能处理玩家对规则的提问，也能在“理解了/继续”后进入下一步。
- `_continue_opening_drama()` 有戏剧化开场演绎，业主出场、孩子失踪、报酬冻结等冲突能建立氛围。
- `_continue_opening_reading()` 会朗读背景与公开导入。
- `_continue_opening_tasks()` 会说明三道最终任务，并点名第一位玩家开始讨论。
- TTS/BGM/SFX 已经在开场节点联动，能增强仪式感。

不足与风险：

- 玩家确认规则后，系统会把戏剧演绎、背景朗读、任务陈述、正式开始连续输出，文本较长，TTS 播放时可能显得像一次性念稿。
- 缺少“玩家准备检查”：例如确认四位角色是否都拿到剧本、是否理解自己的秘密、是否准备进入角色。
- 缺少分段节奏：真人 DM 往往会在规则、背景、任务之间短暂停顿，而当前是一次性推进。
- 没有针对不同玩家新手程度的开场模式，例如快速模式 / 新手模式。
- 没有在开场阶段主动提醒玩家“之后普通讨论时 DM 会少说话，卡住才介入”，玩家可能误以为 DM 会一直陪聊。

建议优化：

- 将开场改为可配置分段：规则确认 -> 氛围演绎 -> 背景朗读 -> 任务确认 -> 第一位发言。
- 每段之间允许玩家输入“继续”，或者在自动模式下短暂停顿。
- 增加“新手提醒”：如何发言、如何隐藏秘密、何时可以问规则、何时不要索要真相。
- 增加“DM 控场声明”：正常讨论时 DM 会保持安静，只有卡顿、偏题、线索公开、阶段切换时介入。

### 重要功能二：流程主持

目标：系统应能稳定带完整局，从开场、讨论、搜证、投票到结局收束，保证各阶段切换清楚、节奏不乱。目标是让玩家感觉有人在主持整场游戏，而不是只能靠手动命令一点点推进。

当前完成度：约 70%，能带完整局，但很多关键推进仍依赖手动命令或固定时间。

已经做好的部分：

- 已有清晰状态机：`opening / opening_rules / discussion / vote / owner_confrontation / postgame_review`。
- 开场能自动进入 discussion，并点名第一位玩家。
- 讨论阶段能自动记录轮次、默认发言人、输入判定、推进信号、案件记忆。
- 线索可以手动 `/clue1`、`/clue2` 公开，也可以按讨论时间自动公开。
- 搜证系统讨论 10 分钟后自动开放，支持 `/search [地点]`。
- 搜证成功会记录 `found_evidence` 和搜索历史，并在主循环中触发一次 search SFX 与 search BGM。
- `/vote` 进入最终公开答案阶段，按固定顺序录入四位角色答案。
- 投票结束后会进行真相揭晓，再进入业主对峙。
- 业主对峙会收集玩家解释，判断诚实/隐瞒并触发 Happy/Bad Ending。
- 结局后进入 `postgame_review`，支持 `/review` 和自由追问。
- `status` 可以查看当前阶段、默认发言玩家、线索、搜证、答案进度、轮次等。
- BGM/SFX/TTS 已接入阶段切换，使流程更像主持而不只是文本系统。

不足与风险：

- 进入投票目前主要依赖玩家手动输入 `/vote`，系统不会根据进度自动建议“可以收束了”。
- 自动公开线索只按真实时间，不看玩家是否已经讨论到相关问题。玩家推理很快时可能等线索，玩家很慢时可能被线索打断。
- 搜证系统和公开线索存在语义映射需要复核：部分搜索结果和 `clue_1 / clue_2` 的公开文本容易让编号含义交叉。
- 阶段切换清楚，但“为什么现在切阶段”的主持解释还不总是自然。
- 讨论阶段仍然强依赖模型控制标记，模型如果输出不稳定，点名、沉默、推进信号会受影响。
- 缺少“流程健康度判断”：例如玩家已经找到两个证据并讨论充分时，系统应建议投票；玩家 20 分钟仍没触及核心问题时，系统应轻推。

建议优化：

- 新增 `ProgressTracker`：综合时间、已公开线索、已发现证据、玩家陈述、关键问题覆盖率，判断“正常推进 / 偏慢 / 卡住 / 可收束”。
- 新增 `GuidancePolicy`：只有在偏慢或卡住时介入；最近刚发过提示、刚公开线索、玩家有突破时不介入。
- 新增 `HintLadder`：同一问题从氛围提示、注意力提示、问题提示到较明确提示逐级递进。
- 新增“建议投票”节点：当两条线索已公开、关键问题覆盖率较高、讨论时间超过建议阈值时，DM 提示“如果没有新的信息，可以准备进入最终公开答案阶段，输入 /vote”。
- 优化 `/status` 为主持视角：不仅显示状态，还显示“下一步建议”。

### 当前最需要大力改善的方向

优先级 P0：智能引导策略

- 建立线索注意力模型：记录玩家是否提到“合页”“钥匙”“橱柜”“碎布”“孩子”“时间线”等核心点。
- 建立玩家提及历史：把 `player_claims_history` 从单纯文本列表升级为结构化实体记录。
- 建立忽略线索检测：公开线索后若多轮没有人提到相关实体，则轻微引导。
- 建立引导冷却：避免连续多轮 DM 插话。
- 将 `_generate_stall_guidance()` 从固定模板改为根据当前推理缺口生成低剧透问题。

优先级 P1：时间与流程健康度

- 增加完整游戏计时，不只 discussion 分钟。
- 增加阶段预算和建议节奏。
- 增加“可正常时间内完成”的判断：例如 30 分钟前是否已经公开两条线索、是否已发现证据、是否开始形成嫌疑链。
- 读档后不要把离线时间算入讨论时长。

优先级 P1：开场分段与主持节奏

- 将当前一次性长开场拆成短段。
- 支持新手模式和快速模式。
- 在开场明确说明 DM 的低干预原则。

优先级 P2：流程自动收束

- 在讨论充分时建议 `/vote`。
- 在投票后把答案检查、真相揭晓、业主对峙之间的文字节奏继续细化。
- 复盘阶段可继续增强为“先总览，再按玩家追问展开”的互动式复盘。

### 是否建议现在做“允许读取陌生脚本”的大改动

不建议立即把主要精力转向陌生脚本初稿。

原因：

- 当前固定剧本主持链路虽然已可演示，但智能引导、进度健康度、自动收束还没有打磨好。
- 陌生脚本会放大所有问题：线索识别、角色秘密、时间线、真相逻辑、提示尺度、复盘结构都需要自动抽取。如果固定剧本下的引导策略还不够稳，换陌生脚本后会更难评估错误来自“脚本解析”还是“主持策略”。
- 更稳的路线是先在固定剧本上把 `ProgressTracker + GuidancePolicy + HintLadder` 做扎实，再抽象成陌生脚本通用接口。

建议路线：

1. 先完善固定剧本的智能引导与流程健康度。
2. 再把固定剧本中用到的结构抽象成通用 schema：角色、公开背景、私密信息、线索、时间线、真相、复盘答案、阶段节奏。
3. 最后做陌生脚本导入的初稿，让 AI 先生成 schema，再由 DM 引擎按同一套主持策略运行。

&#x20;    核心设计原则



&#x20;    1. 听感优先：TTS 清晰度 > BGM 存在感 > SFX 丰富度

&#x20;    2. 无感降级：低库存情绪自动 fallback，玩家无感知

&#x20;    3. 节制触发：SFX 宁可少放，不可扰民（尤其是高频事件）

&#x20;    4. 结局区分：Happy/Bad Ending 必须有不同的音频签名

可以做到，而且这个想法**比单纯“AI NPC 会聊天”更有价值**。你说的“类似 skill 的行动纲领”，本质上就是给 NPC 加一层**可执行的行为策略/行动规则**：它不只是回答问题，而是会根据目标、记忆、当前局势主动选择行为。

这在游戏 AI 里是成熟思路。比如行为树就是用层级节点控制 AI 决策流程，常用于游戏 NPC；GOAP 则是给 NPC 设置目标、行动、前置条件和行动效果，让它根据局势自己规划下一步。([Game Developer][1])  近年的 Generative Agents 也证明了 LLM 角色可以结合记忆、反思和计划，产生更像人的连续行为。([arXiv][2])

你的项目可以做一个简化版，别叫 skill，叫 **NPC Action Profile / 行动纲领** 就行。结构大概是：

```yaml
npc_id: chen_yu
role: 嫌疑人A

goals:
  - 隐瞒自己21:30去过书房
  - 试探玩家是否已经发现钥匙线索
  - 在被怀疑时把注意力引向嫌疑人B

knowledge:
  knows:
    - 自己拿过备用钥匙
    - 受害者生前威胁过自己
  does_not_know:
    - 真凶的完整作案过程
    - 其他角色的隐藏秘密

behavior_rules:
  - if: 玩家询问案发时间线
    then: 给出部分真实信息，但隐瞒21:30去书房
  - if: 玩家出示钥匙线索
    then: 进入紧张状态，承认去过书房但否认杀人
  - if: 玩家连续追问三次
    then: 情绪升级，反问玩家证据是否充分
  - if: 玩家提到嫌疑人B
    then: 主动补充B和死者有矛盾

actions:
  - answer_question
  - deny
  - partially_confess
  - redirect_suspicion
  - ask_player_question
  - reveal_fact_when_exposed
```
我想要做那种ai的npc有类似于skill一样的行动纲领（会比skill简单一点）进行和他人交互，这样可以让玩家有更多的自由度和可玩性，我的项目可以做到类似效果吗？

它和普通 prompt 的区别是：**普通 prompt 只是“你是谁”；行动纲领是“你在什么情况下应该做什么”。** 这会明显提高自由度，因为玩家不是只能按固定选项推进，而是可以随便问，NPC 根据局势反应。

但第一版不要做太复杂。初步设想是：

第一层：**角色知识边界**。NPC 只知道自己的公开信息、私密信息、当前已公开证据和被允许透露的事实。

第二层：**行为规则**。遇到某类玩家输入或某个证据，就触发对应反应，比如否认、转移话题、紧张、部分坦白。

第三层：**行动选择器**。程序先判断当前状态，决定 NPC 本轮应该执行哪个行动，再让 LLM 把这个行动自然地说出来。

也就是说，不要让 LLM 自己决定一切，而是：

**玩家输入 → 程序判断局势 → 选择 NPC 行动 → LLM 生成角色台词。**

这和之前的原则一致：**程序是裁判，LLM 是演员。**

它能从“AI NPC 聊天机器人”升级成：

**“带行动纲领的可控 AI NPC。”**
